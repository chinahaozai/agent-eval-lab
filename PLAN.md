# Agent Eval Lab — 三周落地清单

> 目标：从零做一个「会用工具的 Agent + 轨迹记录 + 三类评测 + 轨迹回放器」的最小可用 Eval 项目，作为转 Agent Eval 方向的求职作品。
>
> 节奏：按业余每天 1–2 小时安排；能集中投入的话两周可压完。每个周末都产出一个「能拿出来给人看」的东西。

## 开工前定三件事

- **项目目录**：本目录独立存在，不要混进其他工程。
- **Provider**：见 `.env.example`，国内推荐 DeepSeek，海外可用 Claude / OpenAI。一套 OpenAI 兼容代码通吃。
- **铁律**：第一个 Agent 必须自己手写循环，**不准用 LangChain**。你做的是 Eval，命根子是看清每一步怎么产生的；框架会把这一切变黑盒。

---

## 第 1 周 · 跑通一个会用工具的 Agent，并把轨迹落盘

> 周末里程碑：终端里 agent 能自己调工具完成任务，每次运行存成一条 JSONL 轨迹。

- [ ] **Day 1 — 第一个 API 调用**：配环境、装 SDK、拿 key，跑通一次对话，打印返回里的 token 数并手算花了多少钱。`【完成标志】`屏幕上有模型回复 + 本次 token 消耗。
- [ ] **Day 2 — 理解 tool use**：定义一个 `calculator` 工具，让模型决定调用它，你手动执行再把结果喂回去拿最终答案。`【完成标志】`能讲清 `messages` 列表是怎么一轮轮增长的。
- [ ] **Day 3 — 手写 agent loop（核心）**：把「模型→工具→模型」包进 while 循环，直到模型不再要求调工具；加 `max_steps` 防死循环。`【完成标志】`一个 `run_agent(task)` 能自动多轮调工具直到给答案。
- [ ] **Day 4 — 加到 3 个工具**：再加 `read_file`、一个简单搜索（真实或 mock）。`【完成标志】`agent 能按任务自己挑用哪个工具。
- [ ] **Day 5 — 定义轨迹结构**：设计 trajectory schema —— task、每个 step（模型输出 / 工具调用入参 / 观察结果）、最终答案、token、成本、耗时。`【完成标志】`跑一次能拿到一个结构化轨迹对象。
- [ ] **Day 6 — 落盘 JSONL + 工程防御**：轨迹序列化成 JSONL（一行一次 run）；给 API 调用加超时和重试。`【完成标志】`磁盘上有 `.jsonl`，能重新读回来。

## 第 2 周 · 评测集 + 三类评测 + 指标

> 周末里程碑：一条命令跑完整个评测集，输出一份指标报告。

- [ ] **Day 7 — 造评测集（关键设计）**：写 15–20 个任务，**刻意分三类**：① 有唯一答案的事实问答 → exact match；② 可计算/可执行验证（数学、代码）→ 可执行校验；③ 开放性（总结、多步）→ LLM-as-judge。每条标注输入、期望答案/验证方式、类型。`【完成标志】`一个 `dataset.jsonl`，每条带评测方式标签。
- [ ] **Day 8 — 批量跑**：写 runner 遍历评测集逐题跑 agent、存全部轨迹；加限速并发和断点续跑。`【完成标志】`一条命令产出 N 条轨迹。
- [ ] **Day 9 — 复盘缓冲**：看轨迹、找 agent 跑挂的地方、修 bug。`【完成标志】`评测集能稳定跑完，数据干净。
- [ ] **Day 10 — 规则类评测**：实现 exact match（含模糊匹配）和可执行校验（数学算结果、代码跑起来对比）。`【完成标志】`规则类任务能自动判对错。
- [ ] **Day 11 — LLM-as-judge**：写 judge prompt，让模型按 rubric 给开放任务打分 + 给理由。`【完成标志】`开放任务能产出分数和判定依据。
- [ ] **Day 12 — 校准 judge（整个项目的灵魂，别跳）**：自己手工标注 5–10 条，和 judge 结果比一致率，调 prompt 提升一致性，亲身体会 judge 的 bias。`【完成标志】`你能说出「我的 judge 和人工标注一致率多少、怎么提上去的」。
- [ ] **Day 13 — 指标汇总**：算分类型成功率、平均步数、平均 token/成本、平均耗时，输出 markdown 表或 json 报告。`【完成标志】`一条命令产出整套指标报告。

## 第 3 周 · 轨迹回放器 + 收尾成作品（你的主场）

> 周末里程碑：一个网页能回放轨迹、看指标，加一份 README，可直接作为求职作品。

- [ ] **Day 14 — 前端脚手架 + 读数据**：Vite + React + TS 起项目，读 JSONL，列出所有 run。`【完成标志】`网页展示 run 列表（任务 / 成功失败 / 成本）。
- [ ] **Day 15 — 轨迹时间线**：点开一条 run，按 step 展示模型思考、工具调用入参、观察结果。`【完成标志】`能在网页上一步步看完一条轨迹。
- [ ] **Day 16 — 调试增强**：高亮失败步、显示每步 token/耗时、工具调用可折叠、把评测结果（对/错/分数+理由）贴到对应 run 上。`【完成标志】`界面上一眼能定位「这条 run 卡在哪步、为什么判失败」。
- [ ] **Day 17 — 指标看板**：加总览页 —— 整体成功率、成本分布、按任务类型的指标，能从总览下钻到单条轨迹。`【完成标志】`打开先看全局指标，再逐层钻下去。
- [ ] **Day 18 — 写成作品**：README 讲清架构和**设计决策**（为什么手写 loop、三类评测怎么分、judge 怎么校准），录一个 30 秒 demo。`【完成标志】`别人 clone 能跑，你能用 3 分钟讲清这项目证明了你什么。

---

## 三条比清单更重要的提醒

1. **Day 12（judge 校准）是杀手锏**。面试时一句「我的 LLM judge 和人工标注一致率是 X，我通过改 rubric 把它从 Y 提到 X」，就甩开只会跑脚本的人 —— 它证明你懂 Eval 的命门是**可信度**，不是代码量。
2. **每天的「完成标志」是防自欺的**。做不到完成标志就别进下一天。
3. **别完美主义**。先把端到端跑通（哪怕丑），再回头打磨。卡某一天超过两小时就记下跳过，周末缓冲日补。
